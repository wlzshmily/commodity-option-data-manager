# Risk Register

| Risk | Impact | Mitigation | Status |
| --- | --- | --- | --- |
| TQSDK live coverage or IV availability varies by product | Incomplete acceptance | Record source unavailable by field group and capture live evidence; do not fail batches when Quote rows are current but Greeks/IV are unavailable | Open |
| Full-market subscriptions overload source/API | Slow or unstable collection | Bounded shards, configurable batch sizes, retry TQSDK startup, batch quote subscriptions with single-symbol fallback, resume successful/interrupted shards, and realtime `quote_stream.contract_months` scope defaulting to the nearest 2 option expiry months with 1/2/3/all choices; 2026-05-09 final local evidence completed 864/864 batches with 4 process shards, 0 failed, `option_batch_size=40`, `wait_cycles=1` | Mitigated |
| Secrets leak through UI/API/test evidence | Security failure | DPAPI storage, masked responses, safe service-log contexts, ignored runtime DBs, tests | Open |
| Existing WebUI baseline may lack approved prototype details | UX mismatch | Preserve current layout and only fill production behavior | Accepted |
| API page status source can block on heavy full-market aggregation | Opening the API page may feel frozen on a large `/mnt/c` SQLite database if it calls `/api/status` | `/#api` now reads lightweight `/api/webui/api-summary` for request metrics, active option count, and latest Quote update, while Swagger/OpenAPI links remain direct | Mitigated |
| Cloud/runtime outbound proxy blocks TQSDK auth | Live validation cannot complete in cloud runtime | Use WSL2/local runtime with direct TQSDK access for acceptance evidence; keep cloud attempts secret-safe and marked blocked | Open |
| WebUI selector state can drift from refreshed overview rows | Wrong T型报价 chain displayed after user changes exchange/product/month | Prefer quote selector rows when available and reload the chain after resolving the changed selector level | Mitigated |
| WebUI T型报价 switching can feel slow on full-market SQLite data | Operator cannot quickly inspect adjacent contracts | Skip selector re-aggregation on contract switches and scope latest K-line lookups to the selected chain | Mitigated |
| Realtime Quote status can be confused with batch collection progress | Operator may expect batch-plan timestamps to advance during realtime subscription | Show realtime worker count/PIDs and latest data write time separately from batch-only progress timestamps | Mitigated |
| Overview underlying table is hard to scan with full-market data | Operator may miss exchange-specific gaps in a long table | Split the table by exchange tabs while keeping the existing all/issue filters | Mitigated |
| Realtime Quote stop can leave stale PID state after process restarts | Operator sees false running state after stopping workers | Validate persisted PIDs by command line and terminate quote-stream process trees when stopping | Mitigated |
| Realtime subscription progress can be mistaken for database write coverage | Operator may see progress stuck below 100% after source fields are empty or after restart | Report progress from worker-created Quote/Kline subscription references, clear stale worker reports on fresh start, and reset progress to zero when stopped | Mitigated |
| Realtime network detection can false-alert during holidays or after close | Operator may restart workers unnecessarily and trigger expensive resubscription | Health detection only evaluates during broad trading-session windows and requires current-session market-time evidence before flagging stale `wait_update` or Quote writes; holidays and post-close periods stay quiet | Mitigated |
| TQSDK reconnect state is opaque during live interruptions | Operator cannot distinguish built-in reconnect from a dead worker and may restart/re-subscribe unnecessarily | Capture official `TqNotify` connection notifications and surface `connected`/`reconnecting`/`disconnected` state in API/WebUI health while leaving TQSDK to auto-reconnect and resend subscriptions | Mitigated |
| New deployments can start realtime subscription before contract discovery | Realtime workers exit because no active Quote/Kline symbols exist in an empty database | `/api/quote-stream/start` now bootstraps the active option universe with one parent-process TQSDK discovery pass before spawning workers, and returns a blocked status with a clear message if discovery fails | Mitigated |
| Pre-session contract refresh or realtime reconciliation is missed | New futures underlyings, new option chains, or new strike-price contracts are not collected at session open, or inactive contracts remain subscribed | Realtime startup now always refreshes contracts, worker 0 periodically refreshes the TQSDK universe during long-running operation, and all workers reconcile Quote/Kline references incrementally without full worker restart | Mitigated |
| Contract refresh timing is not yet exchange/product-specific | Some products may benefit from more exact pre-session windows around night-session and holiday schedules | Current global interval plus startup refresh keeps the universe from going stale during long-running operation; future refinement can add exchange/product-specific calendar scheduling | Open |
| Test server exposes WebUI directly on public `8765/tcp` | Unauthenticated users can inspect data and operate settings/worker endpoints on the test server | Document public binding as temporary test-only in `docs/operations/linux-server-deployment.md`; production access should use SSH tunnel, VPN, Cloudflare Access, or authenticated HTTPS reverse proxy | Accepted for test only |
| Server-side GitHub bare fetch can fail intermittently with TLS transport errors or timeouts | Deployment from `git fetch` may be blocked even when the target GitHub commit exists | For OPS-003, server GitHub `git fetch` failed with `GnuTLS recv error (-110)`; for OPS-004, server `git fetch` timed out connecting to GitHub. In both cases the server successfully downloaded the official GitHub codeload archive for the same commit and built the release from that archive; keep GitHub codeload as the fallback GitHub-sourced path | Mitigated |
| Server deployment can be marked complete before old runtime work is safely stopped and current-slice data is ready | Operators may see realtime subscription progress at 100% while K-line and IV/Greeks rows are absent for later underlyings; stale running batches or a dead metrics worker can remain unnoticed | OPS-005 adds a mandatory Safe Upgrade Gate: stop quote-stream/contract-manager before systemd stop, verify no old worker/WebUI PIDs and no unexpected SQLite holder before switching `current`, restart contract manager/refresh/realtime, verify metrics worker liveness, and block completion on stale running batches plus pending/failed/running current-slice batches unless explicitly recorded as degraded | Open / High priority |
| Runtime operations problems are only found by manual browser inspection | Users may discover K-line/IV/Greeks gaps, stale running batches, or a dead metrics worker before the system tells them | OPS-006 adds code-level operations monitoring: `/api/status` and the WebUI overview now expose `operations` alerts for stale running batches, incomplete current-slice work, metrics-worker not-running/failed states, and subscription-complete rows with missing K-line or metrics rows | Mitigated / Monitor |
| Runtime logs, WAL files, or host resources can exhaust the server | Disk pressure can corrupt operations, block SQLite writes, or make workers fail; CPU/memory pressure can make realtime subscription unstable | OPS-007 adds resource/log monitoring for disk, memory, CPU, SQLite DB/WAL/SHM sizes, runtime log volume, telemetry cleanup status, and SQLite retry events; service logs and API request metrics are retained by age/count, stale rotated logs can be removed, and the deployment gate blocks critical resource states | Mitigated / Monitor |
| SQLite service-state writes can be temporarily locked or lose transaction state during public polling or background work | Realtime status/stop endpoints or request-metric middleware can return 500 even though the business operation succeeded | Service-state writes now rollback on OperationalError, realtime status/stop persistence is best-effort under SQLite busy/locked conditions, and API metrics/log/request-metric writes are best-effort for busy locks plus retryable commit/rollback transaction-state anomalies such as `cannot commit - no transaction is active` | Mitigated |
| TQSDK batch Quote subscription can time out on dense product shards | Realtime worker exits before reaching Kline/running phase, leaving operators with "worker exited" and no active subscription | Batch Quote timeout now falls back to single-symbol `get_quote`, failed symbols are counted/skipped instead of aborting the worker, and test server defaults were reduced to 1 worker with Quote shard size 50 | Mitigated |
| Historical SQLite cache can be mistaken for live realtime data in WebUI | Operator may trust old Quote/Kline rows and believe an unsubscribed chain is current | WEBUI-015 filters overview rows to Quote writes after the current realtime run started and makes T型报价 show `未订阅` until the selected chain has fresh realtime Quote rows | Mitigated |
| Expiring near-month underlying contracts consume limited realtime subscription slots | A 0-day contract such as `pt2606` or `FG606` can prevent the next valid month from being subscribed when the operator chooses nearest 2 contracts | WEBUI-016/017 add `quote_stream.min_days_to_expiry` defaulting to 1, treat option expiry remaining days as authoritative even when the underlying future delivery date is later, and make the overview table use the same subscription scope as realtime Quote/Kline workers | Mitigated |
| Realtime UI state inferred from localized message text | Changing a Chinese prompt can make startup display as stopped or leave buttons in the wrong state | WEBUI-020 persists and returns stable quote-stream lifecycle enums and makes WebUI controls/status cards depend on those enums instead of message text | Mitigated |
| Overview row status can confuse subscription lifecycle with fresh market-data writes | Operators may see progress at 100% while rows still say `订阅中`/`未订阅` if the market has not emitted fresh Quote rows for every option | WEBUI-021 makes row status follow subscription lifecycle enums and keeps Quote/Kline coverage as separate coverage metrics; live verification reached 100% progress and 128 overview rows all `已订阅` | Mitigated |
| Overview row status can imply Kline completion before Kline targets are ready | Operators may see `已订阅` for Quote-complete rows while global Kline progress is still incomplete, and assume Quote plus Kline subscription has completed | WEBUI-025 keeps the status as `订阅中` until both row-level Quote and Kline objects are complete, and shows `Quote x/y` plus `K线 x/y` separately. Running-phase closed-session moneyness recalculation no longer expands Kline targets without actual subscriptions | Mitigated |
| Underlying `last_price` can be empty for closed/no-night contracts | Moneyness-filtered Kline target selection can drop to `0/0` or disagree with T型报价 masking even though TQSDK Quote has usable reference fields such as 昨结/昨收 | MM-T018 centralizes the moneyness reference price as `last_price -> bid/ask midpoint -> close/settlement/pre_settlement/pre_close`, persists TQSDK settlement fields in raw Quote payloads, uses the same helper in realtime Kline selection and T型报价, and verifies worker startup with tuple-row SQLite reads | Mitigated |
| T型报价 and Kline filtering can classify moneyness differently | Operators may see T型报价 colors implying one OTM/ATM boundary while realtime Kline targets use another boundary or omit the underlying Kline from expected row totals | MM-T019 centralizes option-chain moneyness classification in `moneyness.py`; T型报价 now renders backend-provided `moneyness`/`is_atm` fields while realtime Kline filtering consumes the same classification context. AP610 was verified as CALL OTM `24`, PUT OTM `12`, plus underlying Kline total `37` | Mitigated |
| Overview CALL/PUT columns can imply full-chain counts are subscribed targets | Operators may interpret full-chain CALL/PUT counts as moneyness-filtered realtime Kline targets, causing AP610-like rows to show `37/37` when only CALL `24` and PUT `12` options are in the OTM Kline target set | MM-T020 adds per-underlying Kline CALL/PUT target counts to worker progress/API aggregation and uses them in overview rows during realtime operation; the Kline status total still includes the underlying future | Mitigated |
| Theoretical trading-time ranges can mark weekend nights as tradable | Running-phase sticky Kline recalculation can add Kline objects on Saturday night if it only checks time-of-day ranges such as `21:00-02:30` | Trading-session helpers now gate day and night ranges by weekday: Saturday night is out of session, while Saturday early morning can still continue Friday night. Live local Saturday-night observation showed 10 recalculations with 0 sticky additions after the fix | Mitigated |
| Kline setup can block visible T型报价 freshness | Operators may see `已订阅` but T型报价行情时间 and盘口 stay static until every Kline object is created | WEBUI-022 bulk-writes Quote snapshots after Quote subscription, refreshes Quote snapshots every 5 seconds during Kline setup, and keeps startup writes out of the metrics dirty queue so realtime UI data advances before Kline setup completes | Mitigated |
| Product trading sessions differ inside the same exchange | Moneyness recalculation or WebUI trading-time display can be wrong if exchange-level session assumptions are used, e.g. CZCE glass/soda ash night trading versus apple no night trading | SPEC-002 implements product/contract-level Session Manager behavior from each underlying Quote `trading_time`; exchange-level activity is not used as a product proxy | Mitigated / Trial observation |
| TQSDK theoretical `trading_time` can be incomplete during live Quote updates | Overview/T型报价 or moneyness recalculation may mark an active night-session contract as `休市` if `trading_time.night` is empty despite Quote time advancing | Trading-session state now treats fresh underlying Quote source time as authoritative live evidence and uses theoretical `trading_time` only as fallback when Quote time is stale or absent; tests cover fresh versus stale Quote override behavior | Mitigated / Monitor |
| Stale cached `trading_time` can be mistaken for current realtime state before Quote subscription completes | Operators may see `休市` for a row whose current realtime Quote has not been subscribed or written yet, even though the startup state is unknown | Realtime overview shows `待行情` while a row is still pending/subscribing, but shows `休市` once that row is `已订阅` and still has no underlying Quote time | Mitigated |
| Sticky intraday Kline filtering can grow during strong trend days | Kline subscriptions may accumulate beyond the current moneyness match set, reducing but not eliminating resource pressure | Quote remains broad, Kline filtering is sticky only for stability, current match count and cumulative sticky subscribed count are displayed separately; contract-month scope still bounds the universe | Mitigated / Monitor |
| Realtime worker complexity can accumulate into hard-to-maintain code | New filtering/session/logging paths can degrade readability and make later operations risky | Moneyness and trading-session logic live in focused helper modules, realtime worker integration remains aggregate/progress oriented, and default all-range rollback behavior preserves prior scope | Mitigated |
| Moneyness filtering may cause delayed regressions in adjacent realtime features | Quote freshness, metrics refresh, SQLite write behavior, or WebUI responsiveness may degrade only after a live run | Browser-review startup issue where Kline target showed `0/0` was fixed and locally verified; a fresh live session on the new code showed 4 workers advancing through `kline_subscribing` with changing `current_kline_symbol` values and no timeout hits, but longer soak monitoring is still prudent | Mitigated / Monitor |
