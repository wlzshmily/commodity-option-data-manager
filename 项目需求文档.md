# 项目需求文档

## 1. 文档信息

- 项目名称：01.期权数据管理工具
- 文档日期：2026-05-08
- 文档性质：项目需求整理稿
- 需求状态：部分批准

## 2. 项目背景

本项目不是一个通用行情采集器，也不是历史数据仓库。它是未来“期权比例价差监控平台”的数据支持服务，负责采集、管理和展示中国大陆商品期权当前数据切片，为未来监控平台提供可靠的当前合约、行情、K 线、Greeks 和 IV 数据基础。

## 3. 项目目标

项目目标是在本地构建一个小型期权数据管理工具，能够：

- 批量采集中国大陆商品期权当前数据。
- 维护 SQLite 中的最新数据切片。
- 支持未来期权比例价差监控平台读取数据。
- 提供本地 WebUI，让操作人检查采集覆盖、数据新鲜度、异常和当前合约数据。
- 提供轻量本地 API，为未来监控平台集成预留接口。
- 通过严格的 Harness 单元门禁、测试、审计和验收证据完成交付。

## 4. 用户与使用场景

### 4.1 主要用户

- 未来期权比例价差监控平台：作为数据消费者读取当前市场数据。
- 项目 Sponsor / 操作人：通过 WebUI 检查数据采集是否健康、是否覆盖目标市场、是否存在缺失或异常。

### 4.2 核心使用场景

- 操作人查看整体采集状态，判断市场数据是否已经覆盖目标交易所、品种、标的和期权链。
- 操作人进入 T-Quote 页面，查看某个标的期货对应的 CALL / PUT 期权链。
- 操作人查看采集运行记录、失败、重试和回补状态。
- 操作人在设置页配置 TQSDK 账号和密码，并测试连接。
- 未来监控平台通过本地 API 或 SQLite 当前切片读取期权数据。

## 5. 范围定义

### 5.1 当前产品范围

本项目第一阶段应覆盖：

- 中国大陆商品期权市场，包含已批准的一期交易所和商品期权产品。
- 所有可用商品期权品种、所有可用标的期货、CALL 和 PUT 合约。
- 当前必要数据切片，不保留历史快照仓库。
- TQSDK 作为一期数据源。
- SQLite 作为本地存储。
- 本地 WebUI 作为数据健康和行情查看界面。
- 轻量本地 API 作为未来监控平台集成入口。

### 5.2 明确不在范围内

- 实盘交易、下单、撤单。
- 投资建议。
- 期权比例价差机会计算、策略评分、信号、告警和最终策略结果。
- 回测引擎。
- 历史数据仓库。
- 未经 Sponsor 批准的付费数据源或外部数据供应商集成。
- 第一阶段 WebUI 手工逐合约修复 K 线。
- 第一阶段单独市场覆盖页面；覆盖信息应归入 Overview。

## 6. 数据采集需求

### 6.1 数据源

- 一期数据源为 TQSDK。
- 采集服务应使用 TQSDK live reference 模型：Quote 和 K 线引用创建后保持存活，通过 `api.wait_update()` 驱动更新。
- 生产采集循环不得依赖反复创建 Quote / K 线对象的轮询方式。
- 写入 SQLite 应通过 `api.is_changing(...)` 或受控写入间隔作为持久化门控。

### 6.2 采集对象

必须采集和维护：

- 合约元数据：交易所、品种、标的期货、期权合约、CALL / PUT、行权价、到期信息、合约乘数、最小变动价位、活跃状态等。
- Quote 当前数据：最新价、买一价、卖一价、买一量、卖一量、成交量、持仓量、开高低收、均价、源时间等。
- 20 日日 K 线：标的期货和期权合约都应采集，以支持 IV 计算。
- Greeks 和 IV：存储 TQSDK 直接提供或 SDK 工具计算得到、且未来计算需要的数据。
- 采集状态：采集开始/结束、成功/失败/部分不可用、写入数量、错误数量、错误明细等。

### 6.3 当前切片策略

- 第一阶段仅保留最新必要数据切片，不构建历史快照。
- 刷新时应覆盖或替换当前切片。
- 过期、消失或不再可用的合约不得删除，应保留并标记为 inactive。
- 合约宇宙是动态的：新上市合约应变为 active，过期或缺失合约应变为 inactive 或 stale。

### 6.4 数据质量与缺失处理

- 系统必须检测空结果、重复合约、畸形合约记录、网络失败、源 API 格式变化、缺失交易日和陈旧数据。
- 当整体市场开放但某个品种处于非交易或静态状态时，Quote 深度、Greeks 等字段可能为空。系统应按字段组记录源不可用或静态状态，而不是笼统判定整批采集失败。
- Quote、Greeks / IV、K 线各自必须保留源时间戳和应用接收时间 `received_at`。
- 如果新观测到的字段组为空，但此前存在有效值，可以为下游连续性保留 latest effective value，但必须记录 retained 元数据，不能静默伪装成新鲜值。
- 字段组质量应分别记录 Quote trade、Quote depth、Greeks、IV、daily K-line。

### 6.5 批量、分片与恢复

- 全市场或大范围采集必须支持批量/分片，批量大小是性能和压力控制手段，不得减少市场覆盖。
- 中断或失败后，应通过 `collection_plan_batches` 等状态表支持从非陈旧进度恢复。
- 失败分片应隔离记录，后续分片不应因单个分片失败而丢失已完成进度。
- 重试应能跳过已成功分片，只重试 pending / failed 分片。
- 采集压力和批量阈值必须基于实测证据，不得依赖记忆或推断。

## 7. 阶段 rollout 需求
- 覆盖完整 SHFE 铜 `cu` 当前活跃标的和期权链。
- WebUI 选择某个到期月份或标的只影响展示，不得收窄采集范围。
- 必须通过 bounded shards / batches 实现全品种覆盖，不能使用一次性超大订阅集合。
- 设计必须考虑未来全市场效率，避免铜品种专用捷径。

## 8. WebUI 需求

WebUI 已有 Sponsor 批准的交互原型，产品实现应尽可能高保真复现批准原型。未经批准，不得改变用户可见布局、控件、工作流或信息层级。

### 8.1 总体风格

- 专业金融终端风格。
- 浅色、紧凑、数据密集、工作台式布局。
- 以数据表、状态面板和清晰边界为主，避免营销式大留白、装饰背景和大型 hero。

### 8.2 Overview 页面

Overview 是采集健康驾驶舱，应显示：

- TQSDK 连接状态。
- 最新刷新时间。
- 活跃标的数量、活跃期权数量。
- Quote、Greeks / IV、20 日 K 线覆盖。
- 异常/缺失数量。
- 本地 API 健康状态。
- 按交易所的覆盖摘要。
- 按标的期货粒度的主表，而不是按单个期权合约粒度展示。

主要流程：Overview -> 找到异常或目标标的 -> 进入 T-Quote -> 查看完整 CALL / PUT 链和原始数据切片。

### 8.3 T-Quote 页面

T-Quote 是核心市场数据页面，必须满足：

- 一次查看一个选中标的期货的期权链。
- 顶部工具栏包含交易所、品种、到期/月合约选择器。
- 选择器为级联行为：先交易所，再品种，再到期/月。
- 选择器只改变读取/展示模型，不改变后台采集宇宙。
- 中间列为行权价，左侧为 CALL，右侧为 PUT。
- CALL 和 PUT 按行权价对齐。
- 行权价按降序显示，高行权价在上方。
- 标的 bid / latest / ask 显示在 T-Quote 表头区域，与行权价轴视觉关联。
- 近 ATM 行保持中性背景，ITM / OTM 使用 CALL / PUT 侧不同背景色区分。
- IV 和 Greeks 显示固定三位小数；缺失值显示正常缺失标记，不得合成。
- Quote 最新价为空时，静态对比场景可用最新日 K 线收盘价展示 latest；Quote 成交量为空或为 0 时，可用最新日 K 线成交量展示 volume；bid/ask 不得合成。
- 可点击行权价行打开 CALL / PUT 原始数据详情面板。

### 8.4 Acquisition Runs 页面

应显示：

- 采集运行历史。
- 阶段进度。
- 自动重试/回补状态。
- 事件级证据。
- 错误表。

该页面用于诊断和追溯，不用于人工逐合约修复。

### 8.5 API 页面

应显示：

- 本地 API 健康、地址、版本、请求数、延迟、错误率和数据新鲜度。
- API Key 管理：创建、指纹、scope、启用/禁用、最后使用时间、吊销。
- Swagger UI 文档入口。

一期 API 为只读或读为主，不应展示完整 OpenAPI JSON 作为独立入口，除非有明确用户工作流。

### 8.6 Settings 页面

应支持：

- 配置 TQSDK 账号和密码。
- 测试连接。
- 保存凭据到本地 SQLite。
- 保存后账号可见、密码脱敏。
- 配置刷新间隔、自动重试、K 线回补、inactive 处理、并发、SQLite 路径、本地 API bind / port。
- 凭据不得出现在日志、截图、API 响应或源代码中。

## 9. 本地 API 需求

轻量本地 API 面向未来监控平台集成。候选端点包括：

- `GET /api/health`
- `GET /api/status`
- `POST /api/refresh`
- `GET /api/exchanges`
- `GET /api/products`
- `GET /api/underlyings`
- `GET /api/options`
- `GET /api/options/{symbol}/quote`
- `GET /api/options/{symbol}/klines?days=20`
- `GET /api/acquisition-runs`
- `GET /api/acquisition-errors`
- `GET /api/settings`
- `PUT /api/settings/tqsdk-credentials`
- `POST /api/settings/test-tqsdk-connection`

具体 endpoint 集合仍属于待确认项，应在后续单元门禁中批准后实现。

## 10. 数据存储需求

SQLite 数据模型应至少覆盖以下表或等价结构：

- `instruments`：合约和标的元数据、活跃状态、生命周期信息。
- `quote_current`：当前 Quote 数据。
- `kline_20d_current`：标的期货和期权合约的 20 日日 K 线当前切片。
- `option_source_metrics_current`：Greeks、IV 和 SDK 分析数据。
- `acquisition_runs`：采集运行汇总。
- `acquisition_errors`：采集错误明细。
- `collection_plan_batches`：大范围采集批次/分片状态。
- `service_state`：服务运行状态键值。
- `app_settings`：本地设置和敏感配置。

敏感信息必须通过保护措施存储。当前项目已记录：TQSDK 凭据存储在本地忽略的运行数据库中，密码使用 Windows DPAPI 保护；跟踪的 QA 证据数据库不得包含真实凭据。

## 11. 安全与隐私需求

- TQSDK 账号和密码只能通过设置页配置和保存。
- 密码必须脱敏显示。
- 原始密码不得写入日志、状态面板、截图、API 响应、测试证据数据库或源代码。
- API Key 只在创建时完整展示，之后仅展示脱敏指纹。
- 一期本地 API 是否需要认证仍需确认；在确认前不得扩大暴露面。

## 12. 非功能需求

### 12.1 可恢复性

- 采集任务应支持断点恢复。
- 失败批次应可重试。
- 重新发现市场后，陈旧批次应被标记并排除。

### 12.2 可扩展性

- DEV-031B 虽然只覆盖 SHFE 铜 `cu`，但采集架构必须可扩展到 SHFE 和完整商品期权市场。
- 批量大小、引用生命周期、写入节奏、持久化策略和进度报告必须基于实测证据可调。

### 12.3 运行环境

- 项目环境策略为 `uv`。
- 开发单元测试应在 `uv run` 下通过。
- 本地 WebUI 和 API 均为本地运行优先。

## 13. 当前未决事项

以下事项尚未完全冻结，应在后续需求澄清或单元计划中确认：

- 完整目标市场下 IV 来源的可靠性和覆盖表现。
- 初始刷新/写入间隔应偏向低延迟还是低 TQSDK/API 压力。
- WebUI 是否需要手动刷新能力。
- 本地 API 的最终 endpoint 集合。
- 一期 API 是否本地无认证，或需要 API Key 强制认证。
- DEV-031B Sponsor acceptance / signoff 尚未记录，DEV-031C / DEV-031D 不得开始。
